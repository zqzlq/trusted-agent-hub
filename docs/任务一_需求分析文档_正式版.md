# 任务一：可信 Agent Skills 与 MCP Hub 平台需求分析文档

> 调研支持：B、A  
> AI 辅助撰写：Codex（GPT-5）  
> 文档性质：任务一正式需求分析交付物

## 1. 引言

### 1.1 项目背景

随着 Claude Code、Cursor、OpenAI Agents SDK 等智能体工具快速发展，Agent Skills、MCP Server、Plugin、Subagent、Command、Prompt 等能力单元逐渐成为智能体生态中的基础组成部分。用户可以通过这些能力包扩展智能体的工作流、工具调用范围和任务处理能力。

但当前生态存在一个共同问题：能力包的“可发现性”正在增强，而“可信性”并没有同步建立。很多能力包可以通过 GitHub、marketplace、`npx` 或社区目录直接安装，用户能快速得到能力，却很难判断它是否安全、是否越权、是否会读取凭据、是否存在危险命令、是否会访问外部网络。

因此，本课题需要设计并实现一个面向 Agent Skills 与 MCP Server 的可信分发平台，即可信 Skill Hub / MCP Hub。该平台不是普通的下载站，而是一个集浏览、搜索、提交、自动扫描、人工审核、信任评分、安装、更新、下架和审计于一体的治理平台。

### 1.2 文档目标

本文档用于明确任务一阶段的需求边界，输出可供后续任务二至任务七继续实现的统一需求基线。文档重点回答以下问题：

- 平台要服务哪些人。
- 平台要管理哪些对象。
- 平台需要支持哪些核心流程。
- 平台需要提供哪些功能。
- 平台对安全、可追溯性、性能和可扩展性的要求是什么。

### 1.3 术语定义

| 术语 | 定义 |
|---|---|
| Skill | 通过 `SKILL.md` 和相关资源描述的智能体能力单元 |
| MCP Server | 通过 MCP 协议向智能体提供工具、资源或服务的能力单元 |
| Plugin | 聚合多个能力组件的复合包 |
| Subagent | 面向特定任务的专用子智能体 |
| Command | 用户可直接触发的快捷命令 |
| Prompt | 可复用的提示模板 |
| Hub | 可信能力包的浏览、审核、评分和安装平台 |
| 信任评分 | 基于来源、扫描、审核、权限和反馈形成的可解释分数 |
| 下架 | 将存在风险的能力包从可用列表中撤出并停止推荐 |
| 审计日志 | 记录关键操作的不可篡改历史信息 |

### 1.4 范围说明

本文档只描述任务一阶段的需求，不展开实现代码细节，但会明确后续实现必须满足的约束。本文档覆盖：

- Web Hub 浏览与详情。
- 能力包提交与审核。
- 自动扫描与风险提示。
- 信任评分与解释。
- `npx` CLI 安装与校验。
- 版本更新、下架与审计。

不包含：

- 具体前端视觉稿。
- 具体数据库 SQL。
- 具体后端代码实现。
- 生产级高可用部署细节。

## 2. 需求背景与问题分析

### 2.1 现有生态的特点

现有 Skills / MCP 生态已经具备了“发布”和“发现”的基础能力：

- Skill 通过 `SKILL.md` 定义任务流程。
- Plugin 通过 manifest 聚合多种能力。
- MCP Registry 通过元注册表方式索引服务。
- GitHub 和 Release 提供源码与版本托管。
- `npx` 提供零安装门槛的 CLI 分发方式。

### 2.2 现有生态的不足

当前生态的短板集中在可信治理：

- 缺少统一元数据规范。
- 缺少强制审核流程。
- 缺少统一的自动扫描能力。
- 缺少可解释的信任评分。
- 缺少安装前风险提示。
- 缺少可追溯审计链路。
- 缺少对自然语言指令风险的系统性识别。

### 2.3 平台解决的问题

本平台需要解决的核心问题是：让用户不只是“找到并安装能力包”，而是“在可理解风险的前提下使用可信能力包”。

## 3. 系统目标与设计原则

### 3.1 系统目标

1. 建立可信能力包统一入口。
2. 支持多来源接入和统一管理。
3. 建立自动扫描 + 人工审核的双层治理机制。
4. 提供可解释的信任评分。
5. 提供 Web + CLI 两种访问方式。
6. 建立完整的审计和下架机制。

### 3.2 设计原则

| 原则 | 说明 |
|---|---|
| 可信优先 | 先审后发，先提示风险再允许安装 |
| 元数据统一 | 不同来源、不同类型能力包都映射到统一 Schema |
| 风险可解释 | 分数必须能解释扣分原因 |
| 审计可追溯 | 关键操作必须留下记录 |
| 权限最小化 | 默认按最小权限原则审核与安装 |
| 多来源兼容 | 允许 GitHub、npm、PyPI、Docker、本地上传等接入 |
| Web/CLI 一致 | Web 和 CLI 使用同一套后端能力与数据 |

## 4. 用户角色与权限

### 4.1 角色定义

| 角色 | 核心职责 | 主要权限 |
|---|---|---|
| 普通用户 | 发现、查看、安装和反馈能力包 | 浏览、搜索、查看详情、安装、更新、卸载、评分、举报 |
| 提交者 | 创建和维护能力包 | 提交、更新版本、查看扫描报告、响应审核意见 |
| 审核员 | 负责人工审核 | 查看 Diff、扫描报告、权限声明并给出审核结论 |
| 管理员 | 平台治理与风险处置 | 下架、标签管理、审计日志查看、用户与角色管理 |

### 4.2 权限边界

| 能力 | 普通用户 | 提交者 | 审核员 | 管理员 |
|---|---|---|---|---|
| 浏览/搜索 | 是 | 是 | 是 | 是 |
| 查看详情 | 是 | 是 | 是 | 是 |
| 提交版本 | 否 | 是 | 否 | 是 |
| 查看扫描报告 | 部分可见 | 是 | 是 | 是 |
| 审核发布 | 否 | 否 | 是 | 是 |
| 下架能力包 | 否 | 否 | 否 | 是 |
| 查看审计日志 | 否 | 否 | 部分 | 是 |

## 5. 业务场景与用例

### 5.1 普通用户场景

- 用户搜索一个适合代码审查的 Skill。
- 用户查看该 Skill 的权限声明和信任评分。
- 用户通过 `npx` CLI 安装到本地 Claude Code 环境。
- 用户发现高风险项后取消安装。

### 5.2 提交者场景

- 开发者从 GitHub 仓库提交一个新的 MCP Server。
- 系统自动解析其 manifest 和来源信息。
- 系统扫描到敏感环境变量读取行为。
- 提交者修复问题后重新提交版本。

### 5.3 审核员场景

- 审核员查看一个待审核能力包。
- 审核员同时看到源码 Diff、扫描报告和权限声明。
- 若风险可接受，则通过；若风险过高，则驳回。

### 5.4 管理员场景

- 管理员接到安全告警后下架一个已发布插件。
- 管理员查看审计日志确认下架操作来源。
- 管理员对该包添加风险标签，避免再次推荐。

| FR-07 | 系统应提供能力包详情页，展示完整基础信息 |

| FR-11 | 系统应展示审核状态、扫描结果摘要和信任评分解释 |
| FR-12 | 系统应展示版本历史、更新记录、安装统计和用户反馈 |

### 6.3 提交与导入

| 编号 | 需求 |
|---|---|
| FR-13 | 系统应支持从 GitHub 仓库导入能力包 |
| FR-14 | 系统应支持从 GitHub Release 导入压缩包 |

| FR-16 | 系统应支持本地压缩包上传和人工录入 |
| FR-17 | 系统应支持识别不同类型能力包的最小提交规范 |
| FR-18 | 系统应在提交后生成明确的提交状态并可供追踪 |
| FR-19 | 系统应允许提交者对新版本重新提交并保留历史记录 |

### 6.4 自动扫描

| 编号 | 需求 |
|---|---|
| FR-20 | 系统应对每个提交版本执行自动扫描 |
| FR-21 | 系统应执行结构校验，检查关键文件是否存在且格式正确 |
| FR-22 | 系统应执行元数据校验，检查名称、版本、作者、许可证、入口等字段 |
| FR-23 | 系统应扫描提示注入、欺骗性指令和隐藏行为 |
| FR-24 | 系统应扫描危险 Shell 命令、远程代码下载执行和外部网络调用 |
| FR-25 | 系统应扫描敏感环境变量读取、硬编码密钥和越权文件访问 |
| FR-26 | 系统应扫描依赖风险，包括未锁定版本、可疑 npm / pip 依赖和已知漏洞 |
| FR-27 | 系统应输出机器可读的扫描报告，并标注风险等级、位置和建议 |

### 6.5 人工审核

| 编号 | 需求 |
|---|---|
| FR-28 | 系统应提供审核队列，显示待审核、审核中、已完成条目 |
| FR-29 | 系统应允许审核员查看提交内容、Diff、扫描报告和权限声明 |
| FR-30 | 系统应支持通过、驳回、要求修改三种审核结论 |
| FR-31 | 系统应记录每次审核的结论、意见、时间和操作人 |
| FR-32 | 系统应在审核通过后允许版本发布，审核驳回后禁止发布 |

### 6.6 信任评分

| 编号 | 需求 |
|---|---|
| FR-33 | 系统应为每个版本生成 0-100 分信任评分 |
| FR-34 | 系统应支持按来源可信度、作者信誉、元数据完整性、权限最小化、扫描结果、人工审核、版本稳定性、用户反馈和签名可追溯性加权计算 |
| FR-35 | 系统应输出评分解释，说明主要扣分原因和风险点 |
| FR-36 | 系统应在新版本发布、重新扫描和用户反馈变化时更新评分 |

### 6.7 CLI / NPX 安装与管理

| 编号 | 需求 |
|---|---|
| FR-37 | 系统应提供可通过 `npx` 运行的 CLI 工具 |

### 6.9 下架与审计

## 7. 非功能性需求

### 7.1 安全性需求

| 编号 | 需求 |
|---|---|
| NFR-01 | 所有提交版本必须经过自动扫描 |
| NFR-02 | 高危扫描项默认阻断发布 |
| NFR-03 | 安装前必须展示权限声明和风险提示 |
| NFR-04 | 敏感字段和密钥在日志与前端展示中必须脱敏 |
| NFR-05 | 审计日志应具备不可随意修改的存储策略 |
| NFR-06 | 默认采用最小权限原则进行评分和审核 |

### 7.2 可追溯性需求

| 编号 | 需求 |
|---|---|
| NFR-07 | 每个版本必须保存来源、hash、manifest、扫描报告、审核记录和评分解释 |
| NFR-08 | 每次审核和下架都必须保留操作人和时间信息 |
| NFR-09 | 安装记录应可用于后续校验和回溯 |

### 7.3 性能需求

| 编号 | 需求 |
|---|---|
| NFR-10 | 搜索结果应在可接受时间内返回，建议常规查询不超过 2 秒 |
| NFR-11 | 详情页查询应快速返回，建议常规查询不超过 1 秒 |
| NFR-12 | CLI 元数据查询应尽量在 3 秒内完成 |
| NFR-13 | 自动扫描允许异步执行，但应提供可查询的任务状态 |

### 7.4 可用性需求

| 编号 | 需求 |
|---|---|
| NFR-14 | 普通用户应能在不理解内部实现的情况下完成浏览、安装和校验 |
| NFR-15 | 风险解释应使用可读自然语言，而不是只显示黑盒分数 |
| NFR-16 | 高风险操作应有明确的确认和回退提示 |

### 7.5 可扩展性与兼容性需求

| 编号 | 需求 |
|---|---|
| NFR-17 | 系统应支持后续新增能力类型和来源类型 |
| NFR-18 | 系统应支持后续新增扫描规则和评分维度 |
| NFR-19 | 系统应支持后续新增客户端适配方式 |
| NFR-20 | 接口和 Schema 应版本化管理，避免后续扩展破坏兼容性 |

### 7.6 可维护性需求

| 编号 | 需求 |
|---|---|
| NFR-21 | 应提供清晰的 OpenAPI 或接口文档 |
| NFR-22 | 应提供统一的元数据 Schema 和扫描报告 Schema |
| NFR-23 | 应提供明确的状态机定义和错误码规范 |

## 8. 数据需求

### 8.1 核心实体

| 实体 | 用途 |
|---|---|
| users | 用户、角色与权限信息 |
| packages | 能力包主信息 |
| package_versions | 版本、来源、hash、manifest、状态 |
| scan_reports | 自动扫描结果 |
| review_records | 人工审核记录 |
| trust_scores | 信任评分与解释 |
| install_records | 安装统计与本地安装记录 |
| comments | 评论、评分和反馈 |
| audit_logs | 审计日志 |

### 8.2 数据关系要求

| 关系 | 要求 |
|---|---|
| 一个能力包对应多个版本 | 保留历史版本和发布记录 |
| 一个版本对应多个扫描结果 | 支持重新扫描和定期巡检 |
| 一个版本对应多次审核记录 | 支持重审和复审 |
| 一个版本对应多个安装记录 | 支持安装统计和用户反馈 |
| 审计日志独立保存 | 不应依附于普通业务记录 |

### 8.3 关键字段要求

每个版本至少应保存：

- 名称、类型、版本号。
- 来源类型和来源地址。
- 作者信息和许可证。
- 入口文件和适配客户端。
- 权限声明。
- 风险标签。
- 扫描报告索引。
- 审核状态。
- 信任评分。
- 发布时间和下架时间。

## 9. 接口需求

### 9.1 Web 接口需求

Web 前端至少需要支持以下页面：

- 首页与搜索页。
- 能力包详情页。
- 提交页。
- 审核页。
- 管理员页。
- 审计日志页。

### 9.2 后端 API 需求

后端 API 至少需要支持以下能力：

- 能力包创建与版本管理。
- 搜索和详情查询。
- 自动扫描任务提交与结果查询。
- 审核流转。
- 信任评分计算与查询。
- 安装统计与用户反馈。
- 下架和审计查询。
- 登录与权限控制。

### 9.3 CLI 接口需求

CLI 至少需要支持以下命令：

```txt
npx trusted-agent-hub search <keyword>
npx trusted-agent-hub info <name>
npx trusted-agent-hub install <name>
npx trusted-agent-hub update <name>
npx trusted-agent-hub uninstall <name>
npx trusted-agent-hub verify <name>
npx trusted-agent-hub list
```

CLI 输出至少应包含：

- 包名称与版本。
- 审核状态。
- 信任评分。
- 权限声明。
- 风险提示。
- 安装位置。
- 校验结果。

## 10. 状态机要求

### 10.1 提交状态

```txt
draft
→ submitted
→ scanning
→ pending_review
→ approved / rejected / changes_requested
→ published
→ yanked
```

### 10.2 状态说明

| 状态 | 含义 |
|---|---|
| draft | 草稿阶段，尚未提交 |
| submitted | 已提交等待扫描 |
| scanning | 自动扫描进行中 |
| pending_review | 扫描完成，等待人工审核 |
| changes_requested | 审核员要求修改后重提 |
| approved | 审核通过，允许发布 |
| rejected | 审核驳回，不允许发布 |
| published | 已发布，可供浏览和安装 |
| yanked | 已下架，不应再推荐安装 |

## 11. 验收标准

任务一的需求分析文档至少应满足以下验收条件：

1. 明确说明平台背景、目标、角色和范围。
2. 明确列出核心对象、功能需求和非功能需求。
3. 覆盖浏览、搜索、详情、提交、扫描、审核、评分、安装、更新、下架与审计流程。
4. 需求应能直接支撑后续任务二到任务七的实现。
5. 需求中应明确安全控制和风险提示要求。
6. 需求中应明确 Web 与 CLI 两条访问路径。
7. 需求应支持至少 10 个示范能力包的后续建设。

## 12. 风险与约束

### 12.1 主要风险

- 未审核能力包可能包含提示注入和危险脚本。
- MCP Server 可能带来外部网络和数据访问风险。
- 过宽权限会增加用户环境和业务数据的暴露面。
- 来源不清晰会导致版本漂移和供应链风险。

### 12.2 项目约束

- 本课题需兼容多来源接入。
- 本课题需优先保证可信治理主线完整。
- 本课题需在 8 周内完成可演示闭环。

## 13. 结论

可信 Agent Skills 与 MCP Hub 的核心价值，在于把“能力发现”升级为“可信治理”：

- 对外提供统一入口。
- 对内建立自动扫描和人工审核。
- 对用户提供可理解的信任评分和风险提示。
- 对管理员提供下架和审计能力。
- 对后续开发提供统一元数据和接口基础。

只要满足本文档定义的需求，后续实现就可以围绕一条清晰主线展开：

```txt
提交 → 扫描 → 审核 → 发布 → 浏览/搜索 → CLI 安装 → 更新/校验 → 下架/审计
```

这条主线即是本课题后续任务二到任务七的实现基础。
